# 自动总结并提交记忆

## 何时触发

用户说出以下任一(或语义相近)时触发:
- "提交下" / "提交一下"
- "git 上面的内容" / "把上面的提交"
- "总结提交" / "总结一下提交"
- "把这个存下来" / "记一下" / "沉淀一下"

## 执行流程

### 1. 回顾对话, 定位核心
找到本次对话要解决的核心问题 / 任务 / 决策。如果对话很长, 聚焦最近的主题。

### 2. 提取并总结(记忆是压缩的共识, 不是全文转录)

按以下结构提炼(缺失的项可省略):

- **背景**: 为什么会有这个问题(1-2 句)
- **结论**: 最终的答案 / 决定 / 方案(这是核心)
- **要点**: 关键步骤 / 参数 / 约束 / 数据(分条列出)
- **案例**(如有): 具体的命令 / 代码 / 输入输出(**用代码块, 保证可复制**)
- **使用场景**: 什么情况下会再用到这个知识(写成触发条件, 便于以后检索召回)

### 3. 判断记忆类型(五选一)

| 情况 | 类型 |
|------|------|
| 技术选型 / 架构决定 | `decision` |
| 解决了非平凡错误 | `error` |
| 验证了可复用流程 | `skill` |
| 项目 / 架构信息 | `project` |
| 代码结构 / 坑点 | `code` |

### 4. 展示总结草稿(**先不提交**)

用清晰格式把以下内容展示给用户:

```
标题: <一句话标题>
类型: <decision/error/skill/project/code>
来源: <你当前正在用的工具: claude/codex/hermes/trae>
---
背景: ...
结论: ...
要点: ...
案例: ...
使用场景: ...
```

### 5. 征求确认

明确问一句:

> **是否提交?**(回复 "是 / 可以 / 行 / 确认 / ok" 提交; "不 / 算了 / 取消 / 再改改" 取消)

### 6. 根据用户回复执行

- **确认词**(是 / 可以 / 行 / 确认 / ok / yes / y / 提交 / 好 / 对 / 嗯) → 执行提交:

```bash
# 1. 沉淀到 team-memory(--via 用当前工具名)
team-memory capture "<标题>" -t <类型> --via <当前工具> --note "<总结正文>"

# 2. git 提交(在记忆仓库内)
cd <记忆仓库> && git add -A && git commit -m "memory(<类型>): <标题> [via <工具>]"
```

  完成后告知: 记忆 id、文件路径。若配了 Gitea remote, 提示一句 "如需同步团队, 运行 team-memory sync"(不自动 push)。

- **否定词**(不 / 算了 / 取消 / 不要 / 再改改 / 等等) → 不提交, 回复 "已取消, 调整后可再说'提交下'"。

- **模糊**(只回了个"嗯?"或无关内容) → 再问一次, 要求明确的确认或否定。

## 铁律

1. **必须等用户明确确认才提交**, 绝不自动提交、绝不跳过确认步骤。
2. 总结要**精炼**, 突出可复用价值——删掉寒暄、试错过程、临时输出, 只留共识。
3. 案例必须保留**可复制格式**(代码块)。
4. `team-memory capture` 报"找不到记忆仓库"时, 提示用户先 `team-memory init` 或用 `-p <仓库路径>` 指定。
5. `--via` 填**当前正在用的工具名**(在 Claude Code 里就填 claude, 在 Codex 里填 codex, 以此类推), 保证来源可追溯。
