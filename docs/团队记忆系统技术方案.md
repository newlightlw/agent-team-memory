# AI Native 团队记忆系统 — 技术方案

## 一、架构概述

### 1.1 四层记忆模型

我们的记忆架构基于四层模型：

| 层级 | 范围 | 持久性 | 写入时机 |
|------|------|--------|----------|
| 工作记忆 | 当前会话上下文 | 临时（对话窗口内） | 自动 |
| 精选长期记忆 | 团队共享，经过蒸馏验证 | Git 提交的文件（长期） | 任务完成后，经蒸馏写入 |
| 完整历史 | 每个会话的完整记录 | 各 Agent 实例的本地存储 | 自动 |
| 外部知识源 | MCP 服务器、向量数据库、网络搜索 | 外部系统 | 按需召回 |

**核心原则**：

1. **新写入不在当前 session 生效** — 防止写入行为污染当前对话
2. **外部召回不写回历史** — 工具获取的信息用于当前任务，不经蒸馏不持久化
3. **判断信息存哪一层，比存得更聪明更重要**

### 1.2 两期实施策略

| 阶段 | 时间 | 核心能力 | 基础设施 |
|------|------|----------|----------|
| Phase 1 | 第 1-2 周 | 文件级记忆，Git 版本管理，人工审核 | 零新增（仅用现有 Gitea） |
| Phase 2 | 第 3-4 周 | 语义检索，跨工具统一访问 | MCP 服务器（部署在 Gitea 同环境） |

### 1.3 五类记忆映射到存储

| 记忆类型 | Phase 1 存储位置 | Phase 2 增强能力 |
|----------|-----------------|-----------------|
| 项目记忆 | `team-memory/memory/project/` | MCP 知识图谱实体 |
| 代码记忆 | 各项目 `memory/code/` | MCP 向量检索 |
| 决策记忆 | `team-memory/memory/decisions/` | MCP 语义条目 |
| 错误记忆 | `team-memory/memory/errors/` | MCP 自动捕获 |
| Skill 记忆 | `team-memory/memory/skills/` | MCP 过程记忆 |

## 二、目录与文件结构

### 2.1 team-memory 共享仓库

在 Gitea 上创建 `team-memory` 仓库，结构如下：

```
team-memory/
├── CLAUDE.md                              # 团队级 Agent 指令（每次会话自动加载）
├── rules/
│   ├── team-culture.md                    # 团队行为准则（来自宣导文档）
│   ├── code-standards.md                  # 编码规范
│   └── agent-guidelines.md               # Agent 行为规范
├── memory/
│   ├── project/
│   │   ├── product-overview.md            # 产品概述：做什么、给谁用、核心场景
│   │   ├── system-architecture.md         # 系统架构：模块边界、技术栈、数据流
│   │   └── team-roles.md                  # 团队分工：谁负责什么、协作方式
│   ├── code/
│   │   ├── repo-map.md                    # 所有仓库的地图和关系
│   │   ├── shared-libs.md                 # 共享库和 API 契约
│   │   └── deployment.md                  # 部署架构和环境信息
│   ├── decisions/
│   │   ├── adr-001-xxx.md                 # 架构决策记录（ADR）
│   │   ├── adr-002-xxx.md
│   │   └── _index.md                      # 所有 ADR 的索引
│   ├── errors/
│   │   ├── 2026-06-xxx.md                 # 按日期的错误记录
│   │   └── _index.md                      # 错误索引（按类型/模块分类）
│   └── skills/
│       ├── prompt-engineering.md          # 已验证的 prompt 模式
│       ├── model-evaluation.md            # 模型评测方法论
│       ├── badcase-analysis.md            # Badcase 分析流程
│       └── _index.md                      # Skill 索引
├── templates/
│   ├── memory-template.md                 # 通用记忆模板
│   ├── adr-template.md                    # ADR 模板
│   └── error-template.md                  # 错误记忆模板
└── scripts/
    ├── sync-to-local.sh                   # 同步到本地开发机
    └── validate-memory.sh                 # 校验记忆文件格式
```

### 2.2 各项目仓库内记忆

每个项目仓库（如 `intent-engine`、`conversation-flow`）在根目录下增加：

```
project-repo/
├── CLAUDE.md                              # 项目级 Agent 指令
├── .cursorrules                           # Cursor 桥接文件（可选）
├── AGENTS.md                              # Codex CLI 桥接文件（可选）
├── memory/
│   ├── project-context.md                 # 本项目的上下文说明
│   ├── code-structure.md                  # 本项目的代码地图
│   ├── pitfalls.md                        # 本项目已知的坑点
│   └── local-decisions.md                 # 本项目的小型决策记录
├── src/
└── tests/
```

**项目记忆与团队记忆的关系**：

- 项目 `memory/` 存本项目专属知识（代码结构、本地约束）
- 跨项目通用知识（产品背景、通用 skill）存在 `team-memory` 仓库
- 项目 CLAUDE.md 中通过链接引用 team-memory 中的相关条目，避免重复
- Agent 加载时：先加载团队级，再叠加项目级

### 2.3 个人本地记忆

每个开发者的机器上：

```
~/.claude/
├── CLAUDE.md                              # 个人全局指令（已有，保持不变）
├── MEMORY.md                              # 个人自动记忆（Claude Code 原生）
├── settings.json                          # 个人设置
├── rules/                                 # 个人规则（已有）
│   ├── coding-style.md
│   ├── git-workflow.md
│   └── ...
├── skills/
│   ├── team/                              # symlink → ~/projects/team-memory/
│   └── learned/                           # 个人沉淀的 skill
└── projects/
    └── <project>/
        └── memory/
            └── MEMORY.md                  # 项目级个人记忆（Claude Code 原生）
```

### 2.4 Agent 加载顺序

```
会话启动
  │
  ▼
加载 ~/.claude/CLAUDE.md                    （个人全局上下文）
加载 ~/.claude/MEMORY.md                    （个人自动记忆）
加载 ~/.claude/rules/*.md                   （个人规则）
  │
  ▼
如果当前在项目目录中：
  加载 ~/projects/team-memory/CLAUDE.md      （团队共享上下文）
  加载 ~/projects/team-memory/rules/*.md     （团队规则）
  加载 ./CLAUDE.md                           （项目级上下文）
  加载 ./memory/*.md                         （项目记忆）
  │
  ▼
Phase 2 增加：
  MCP 语义检索（按需查询团队记忆）
```

## 三、记忆数据模型

### 3.1 通用记忆格式

每条记忆统一使用 YAML frontmatter + Markdown 正文：

```markdown
---
id: mem-2026-0611-001
type: project | code | decision | error | skill
scope: team | project | module
author: 张三
role: algorithm | data | product | fullstack
created: 2026-06-11
updated: 2026-06-11
expires: null
status: active | deprecated | superseded-by:mem-2026-XXXX-XXX
tags: [nlp, 意图识别, 医美]
related: [mem-2026-0601-003]
---

# 标题

## Context（背景）
为什么需要这条记忆 — 什么场景触发了它

## Content（内容）
实际的知识内容（格式因类型而异，见下方）

## Validation（验证）
如何确认这条记忆是正确的
```

### 3.2 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是 | 唯一标识：`mem-YYYY-MMDD-NNN` |
| `type` | 是 | 五类之一：project / code / decision / error / skill |
| `scope` | 是 | 可见范围：team（全团队）/ project（项目内）/ module（模块内） |
| `author` | 是 | 作者名 |
| `role` | 是 | 岗位：algorithm / data / product / fullstack |
| `created` | 是 | 创建日期 |
| `updated` | 是 | 最后更新日期 |
| `expires` | 否 | 过期日期，null 表示永不过期 |
| `status` | 是 | active / deprecated / superseded-by:mem-XXX |
| `tags` | 否 | 标签列表，用于分类检索 |
| `related` | 否 | 关联的其他记忆 ID |

### 3.3 各类型 Content Schema

#### 项目记忆

```markdown
## Content

### 背景
[产品做什么，目标用户，核心业务场景]

### 架构
[系统架构图或文字描述，模块边界]

### 约束
[硬性限制：合规要求、性能 SLA、技术选型约束]

### 模块地图
| 模块 | 负责人 | 仓库 | 状态 |
|------|--------|------|------|
| 意图引擎 | 算法组 | intent-engine | 活跃 |
| 对话流程 | 产品组 | conversation-flow | 活跃 |
```

```markdown
[目录布局，关键文件及其用途]
```

## 四、CLAUDE.md 层级体系

### 4.1 团队级 CLAUDE.md（team-memory 仓库根目录）

这是所有 Agent 会话都会加载的核心文件：

```markdown
# [团队名] Agent 记忆中心

本仓库是团队共享记忆系统。

## Agent 启动指令

在任何项目会话中：

1. 先读本文件
2. 读 rules/ 下的团队规范
3. 如果在具体项目中，读该项目的 CLAUDE.md
4. 做重要变更前，先查 memory/decisions/ 中的相关 ADR
5. 解决非平凡问题后，用 templates/ 中的模板沉淀记忆

## 团队身份

- 产品：AI 驱动的对话系统（话术 Agent OS），面向医疗/医美行业
- 团队规模：12 人
- 岗位：算法、数据分析、产品、全栈

## 五类记忆速查

| 类型 | 位置 | 何时读取 | 何时写入 |
|------|------|----------|----------|
| 项目 | memory/project/ | 接手新任务时 | 项目范围变化时 |
| 代码 | 各项目 memory/code/ | 写代码前 | 发现重要结构时 |
| 决策 | memory/decisions/ | 做技术选择前 | 做出重要决策后 |
| 错误 | memory/errors/ | 遇到错误时 | 解决非平凡错误后 |
| Skill | memory/skills/ | 执行流程前 | 验证可复用流程后 |

## 记忆写入规范

1. 不要在任务执行中写记忆 — 任务完成后再写
2. 使用 templates/ 下的标准模板
3. 填写 author 和 role 字段
4. 添加 related 关联到相关记忆
5. 走 PR 审核（Phase 1 第 2 周起）
6. 废弃不删除 — 更新 status 字段并链接到新记忆

## 仓库索引

- ADR 索引：memory/decisions/_index.md
- 错误索引：memory/errors/_index.md
- Skill 索引：memory/skills/_index.md
```

### 4.2 项目级 CLAUDE.md（各项目仓库根目录）

```markdown
# [项目名]

## 概述
[2-3 句话描述这个服务/模块做什么]

## 架构
[链接到 team-memory 中的系统架构记忆，以及本项目特有的架构说明]

## 关键文件
| 文件 | 用途 |
|------|------|
| src/main.py | 应用入口 |
| src/agent/ | Agent 编排逻辑 |

## 依赖
- [关键依赖 1] — 版本约束，为什么重要

## 测试
[如何运行测试、覆盖率要求]

## 相关记忆
- 项目上下文：memory/project-context.md
- 已知错误：[gitea-url]/team-memory/memory/errors/_index.md
- ADR：[gitea-url]/team-memory/memory/decisions/_index.md
```

## 五、记忆生命周期

### 5.1 创建流程

```
任务完成
  │
  ▼
开发者/Agent 识别值得保留的知识
  │
  ▼
使用模板起草记忆条目
  │
  ▼
判断范围：
  - 个人经验 → ~/.claude/skills/learned/ 或 MEMORY.md
  - 项目专属 → 项目仓库 memory/
  - 团队通用 → team-memory 仓库
  │
  ▼
[团队通用记忆] 提交到 team-memory
  - 第 1 周：直接提交到 main（培养习惯）
  - 第 2 周起：提 PR，至少一人审核
  │
  ▼
合并后 → 下一个会话中 Agent 可读取
```

### 5.2 审核机制（渐进式）

| 阶段 | 审核要求 | 目标 |
|------|----------|------|
| 第 1 周 | 无审核，直接提交到 main | 降低门槛，培养写入习惯 |
| 第 2 周 | PR 提交，至少一人 Review | 建立质量意识 |
| 第 3-4 周 | PR + 跨岗位审核 + CI 校验 | 确保准确性 |
| 稳定期 | PR + 跨岗位审核 + CI + 周审计 | 持续质量保障 |

### 5.3 更新流程

```
发现记忆过时
  │
  ▼
创建新记忆，status: active
更新旧记忆，status: superseded-by:mem-新ID
  │
  ▼
提交 PR（同创建流程）
```

### 5.4 废弃规则

| 条件 | 处理 |
|------|------|
| `expires` 日期已过 | 在周审计中标记 |
| 代码记忆引用的文件已删除 | 下次接触时更新，或在周审计中标记 |
| 决策被新 ADR 取代 | 链接旧到新，保留旧的作为审计记录 |
| 错误记忆超过 6 个月 | 审查是否仍有相关性，否则废弃 |
| Skill 产出错误结果 | 立即废弃，创建 issue 替换 |

## 六、跨工具兼容方案

团队使用多种 Agent 工具，记忆系统需要兼容所有工具。

### 6.1 工具对比与桥接

| 工具 | 原生记忆机制 | 桥接方式 |
|------|------------|----------|
| **Claude Code** | `CLAUDE.md` + `MEMORY.md` + `rules/` + MCP | 原生支持，无需桥接 |
| **Cursor** | `.cursorrules` 文件 | 创建 `.cursorrules` 指向 team-memory 内容 |
| **Codex CLI** | `AGENTS.md` 文件 | 创建 `AGENTS.md` 镜像项目 CLAUDE.md 内容 |
| **其他 MCP 工具** | MCP 协议 | Phase 2 通过 MCP 服务器统一访问 |

### 6.2 Cursor 桥接：`.cursorrules`

在每个项目根目录创建 `.cursorrules`：

```markdown
# Team Memory Bridge (Cursor)

## 团队上下文
读取以下文件获取团队背景：
- ../team-memory/CLAUDE.md
- ../team-memory/rules/team-culture.md
- ../team-memory/memory/project/product-overview.md

## 项目上下文
读取以下文件获取项目背景：
- memory/project-context.md
- memory/code-structure.md
- memory/pitfalls.md

## 记忆写入
完成任务后，使用 team-memory/templates/ 中的模板沉淀记忆。
团队通用记忆提交到 team-memory 仓库，项目记忆提交到本项目。
```

### 6.3 Codex CLI 桥接：`AGENTS.md`

在每个项目根目录创建 `AGENTS.md`，内容与项目 `CLAUDE.md` 保持一致。

### 6.4 同步脚本

```bash
#!/bin/bash
# scripts/sync-to-local.sh — 每日或工作前运行

TEAM_MEMORY_DIR="$HOME/projects/team-memory"
CLAUDE_SKILLS_DIR="$HOME/.claude/skills/team"

# 拉取最新
cd "$TEAM_MEMORY_DIR" && git pull origin main

# 更新 symlink（幂等）
ln -sfn "$TEAM_MEMORY_DIR" "$CLAUDE_SKILLS_DIR"

# 同步 .cursorrules 到活跃项目
for project_dir in ~/projects/*/; do
  if [ -d "$project_dir/.git" ]; then
    cp "$TEAM_MEMORY_DIR/templates/cursorrules-template" \
       "$project_dir/.cursorrules" 2>/dev/null || true
  fi
done

echo "Team memory synced."
```

## 七、团队工作流集成

### 7.1 每日工作循环

```
开始任务前
  │
  ├─ Agent 自动加载 CLAUDE.md（团队 + 项目）
  ├─ 根据任务类型浏览相关记忆：
  │   新功能 → memory/project/ + memory/decisions/
  │   Bug 修复 → memory/errors/
  │   重构 → memory/code/ + memory/decisions/
  └─ 确认 Agent 理解了项目背景和约束
  │
  ▼
执行任务
  │
  ├─ Agent 基于已加载的记忆工作
  ├─ 如果发现值得记录的知识，先记住（不立即写入）
  └─ 关键判断在对话中保留
  │
  ▼
任务完成后（记忆蒸馏）
  │
  ├─ 回顾：这次任务有什么值得保留的？
  ├─ 使用模板起草记忆条目
  ├─ 判断范围（个人 / 项目 / 团队）
  ├─ 提交到对应位置
  └─ 记忆在下一个会话中生效
```

### 7.2 Git Flow 集成

```
main ──────────────────────────────────
  \                                    \
   feat/xxx ──→ PR ──→ merge           team-memory 更新 PR
        |                                |
        | (功能开发中)                    | (功能合并后)
        |                                |
   代码变更在这里                    记忆蒸馏在这里：
   - 正常 commit                     - 做了什么
   - 正常 PR review                  - 为什么这样决定
   - CI 检查                         - 遇到了什么错误
                                     - 用了什么 skill
```

记忆分支命名规范：

```
memory/project/xxx-context
memory/decision/adr-015-xxx
memory/error/xxx-resolution
memory/skill/xxx-procedure
```

### 7.3 PR 模板

在 team-memory 仓库中创建 `.gitea/PULL_REQUEST_TEMPLATE/memory.md`：

```markdown
## 记忆类型
[ ] 项目记忆  [ ] 代码记忆  [ ] 决策记忆  [ ] 错误记忆  [ ] Skill 记忆

## 摘要
[添加或更新了什么知识]

## 来源
[什么工作产出了这个知识 — PR 链接、Issue 链接、事件描述]

## 审核清单
- [ ] 使用了标准记忆模板
```

### 8.3 校验脚本

```bash
#!/bin/bash
# scripts/validate-memory.sh

MEMORY_DIR="$1"
ERRORS=0

for file in $(find "$MEMORY_DIR" -name "*.md" ! -name "_index.md"); do
  # 检查 YAML frontmatter
  if ! head -1 "$file" | grep -q "^---"; then
    echo "ERROR: $file 缺少 YAML frontmatter"
    ERRORS=$((ERRORS + 1))
    continue
  fi

  # 检查必填字段
  for field in id type scope author created status; do
    if ! grep -q "^${field}:" "$file"; then
      echo "ERROR: $file 缺少必填字段: $field"
      ERRORS=$((ERRORS + 1))
    fi
  done

  # 检查 type 是否合法
  TYPE=$(grep "^type:" "$file" | awk '{print $2}')
  case "$TYPE" in
    project|code|decision|error|skill) ;;
    *) echo "ERROR: $file type 不合法: $TYPE"; ERRORS=$((ERRORS + 1)) ;;
  esac
done

if [ $ERRORS -gt 0 ]; then
  echo "校验失败，共 $ERRORS 个错误"
  exit 1
fi

echo "校验通过"
```

### 9.1 架构

```
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐
│        MCP 记忆服务器                 │
│  ┌──────────────────────────────┐   │
│  │  向量存储（语义搜索）          │   │
│  └──────────────────────────────┘   │
│  ┌──────────────────────────────┐   │
│  │  知识图谱（实体+关系）        │   │

└──────────────┬──────────────────────┘
               │ MCP 协议
     ┌─────────┼──────────┐
     ▼         ▼          ▼
 Claude Code  Cursor   Codex CLI
```

**核心原则**：Git 文件是 Source of Truth。MCP 是只读优化层。写入永远走 Git。

### 9.2 MCP 工具列表

| 工具 | 用途 |
|------|------|
| `search_memory` | 语义搜索所有记忆 |
| `get_memory` | 按 ID 获取特定记忆 |
| `get_project_context` | 获取项目的完整上下文 |
| `get_recent_decisions` | 获取最近的 ADR |
| `get_error_solutions` | 按错误模式搜索解决方案 |
| `get_skill` | 获取验证过的 Skill 流程 |

**注意**：不暴露 `write_memory` 工具。写入走 Git PR 审核流程。

### 9.3 MCP 服务器选型

| 方案 | 语义搜索 | 知识图谱 | 部署方式 | 适配性 |
|------|---------|---------|---------|--------|
| **agentmemory** | 是 | 有限 | Docker / 本地 | 支持 Claude Code / Cursor / Codex |
| **omega-memory** | 是 | 是 | uvx / Docker | 功能更全面 |
| **自建** | 定制 | 定制 | 完全自控 | 开发成本高 |

**建议**：先用 agentmemory（部署简单、社区活跃），如不满足再升级到 omega-memory。
