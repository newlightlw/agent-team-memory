# Team Memory 系统说明

> 本文档说明 **team-memory**（团队 Agent 记忆共享系统）的原理、架构、工作流程，
> 以及**何时触发记忆存储**、**何时触发记忆共享**。
> 这是已实现系统的说明，非方案提案。方案提案见同目录另外两份文档。

---

## 一、这是什么

一套把分散在 **Claude Code / Codex / Hermes / Trae** 等 AI coding 工具里的记忆，
统一汇聚到 **Gitea 上的 Markdown 仓库**的系统，让团队成员和所有 AI 工具读同一套上下文。

**解决的问题**：多个 AI 工具各自有记忆（Claude 的 `CLAUDE.md`、Codex 的 `AGENTS.md`、
Hermes 的 Skill、Trae 的 rules），导致**上下文割裂、经验无法复用、新人接手成本高**。

**一句话本质**：

> 用 **Git 管理的 Markdown** 承载团队长期记忆，用 **全局 skill** 让所有工具读同一套上下文，
> 用 **来源/版本字段** 追溯每条记忆从哪来、改过几次。

## 二、核心原理

### 2.1 四层记忆模型

| 层级 | 范围 | 持久性 | 本系统管不管 |
|------|------|--------|--------------|
| 工作记忆 | 当前对话上下文 | 临时（对话窗口内） | 不管（各工具自带） |
| **精选长期记忆** | **团队共享，蒸馏后的共识** | **Git 提交的 Markdown** | **本系统的核心** |
| 完整历史 | 每个会话的完整记录 | 各工具本地（state.db 等） | 不管 |
| 外部知识源 | 向量库、MCP、网络搜索 | 外部系统 | Phase 2 可选接入 |

**核心判断**：一条信息该存哪一层，比"存得更聪明"更重要。本系统只负责**第二层**——
把值得长期保留的团队认知压缩成 Markdown 沉淀下来。

### 2.2 五类记忆

每条记忆必属一类，落到对应目录：

| 类型 | 目录 | 回答的问题 |
|------|------|------------|
| `project` | `memory/project/` | 项目做什么？架构？约束？ |
| `code` | `memory/code/` | 代码结构？入口？坑点？ |
| `decision` | `memory/decisions/` | 哪些技术决策已经定了？（ADR） |
| `error` | `memory/errors/` | 遇到过什么错？怎么解的？ |
| `skill` | `memory/skills/` | 有哪些验证过的可复用流程？ |

### 2.3 三条铁律

1. **Git 是唯一真相** —— Markdown 文件是事实来源；Web/skill 都是从它派生的只读视图。
2. **新写入不在当前会话生效** —— 写入后**下一个会话**才加载，避免污染当前对话。
3. **废弃不删除** —— 记忆过时改 `status: superseded` + `superseded_by` 指向新记忆，旧的留作审计。

## 三、系统架构

### 3.1 组件

```
agent-team-memory/（工具仓库，Python CLI）
├── team_memory/
│   ├── models.py        记忆数据模型(MemoryEntry + frontmatter 解析)
│   ├── store.py         记忆仓库定位 + CRUD
│   ├── config.py        配置(.env → Gitea/团队/作者)
│   ├── gitutil.py       git 操作封装(版本历史靠它)
│   ├── templating.py    模板渲染
│   ├── sources.py       多源聚合(team-memory + tencent-openclaw)
│   ├── bridges/         四工具全局桥接(Claude/Codex/Hermes/Trae)
│   ├── commands/        init/capture/load/sync/list/show/web
│   ├── cli.py           typer 入口(team-memory 命令)
│   └── web/             零依赖 Web UI(http.server + 单 HTML)
└── templates/           记忆模板 + 仓库骨架
```

### 3.2 数据流（一张图看清）

```
                         ┌─────────────────────────────────────┐
                         │     记忆仓库 (Git, 推 Gitea)          │
                         │   memory/{project,code,decisions,    │
                         │            errors,skills}/mem-*.md   │
                         │   每条带 frontmatter + source + body │
                         └──┬──────────┬──────────┬─────────────┘
                            │ 写入     │ 读取     │ 读取
                   capture  │          │          │
         ┌──────────────────▼┐  ┌──────▼─────┐  ┌─▼──────────┐
         │  各 AI 工具会话    │  │ team-memory│  │  Web UI    │
         │  (沉淀记忆)        │  │ list/show  │  │  检索/可视化│
         └────────┬───────────┘  └────────────┘  └────────────┘
                  │ load --global 注册 skill
                  ▼
         ┌────────────────────────────────────────┐
         │ ~/.claude/skills/  ~/.codex/AGENTS.md   │
         │ ~/.hermes/skills/  ~/.trae/skills/      │
         │ (4 工具都能读到同一个记忆仓库)            │
         └────────────────────────────────────────┘
```

## 四、记忆的数据结构

每条记忆 = **YAML frontmatter + Markdown 正文**：

```markdown
---
id: mem-20260624-001              # 唯一 id: mem-YYYYMMDD-NNN（自动生成，全局递增）
type: decision                    # 五类之一
scope: team                       # team / project / module
author: vayne                     # 沉淀者
source: claude                    # ★ 来源工具: claude/codex/hermes/trae
role: fullstack                   # 岗位
created: 2026-06-24               # ★ 入库时间
updated: 2026-06-24               # 最后更新
status: active                    # active/deprecated/superseded
tags: [architecture]              # 检索标签
related: [mem-20260624-002]       # 关联记忆（跨工具引用）
---

# 标题

## Context / Content / Validation  （正文，按类型有模板）
```

**版本** = 该文件的 git 提交历史（`git log -- <file>`），每次 commit 是一个版本，
show 命令和 Web UI 都会展示完整 commit 列表。

## 五、工作流程（日常循环）

```
① 会话启动
   工具加载全局 team-memory skill → 指向记忆仓库 → 知道团队上下文在哪
        │
② 接到任务，先读记忆
   新功能 → 读 memory/project/ + memory/decisions/
   修 Bug → 读 memory/errors/
   执行流程 → 读 memory/skills/
        │
③ 执行任务（用任何工具：Claude/Codex/Hermes/Trae）
   关键判断在对话里保留，但**不立即写入**记忆
        │
④ 任务完成 —— 记忆蒸馏（★ 存储触发点）
   回顾：这次有什么值得保留的？→ team-memory capture（标注 --via 来源）
        │
⑤ 共享给工具和团队（★ 共享触发点）
   team-memory load --global   刷新 4 工具的 skill（让它们看到新记忆）
   team-memory sync             推到 Gitea（让队友拉到）
        │
⑥ 下一个会话
   新写入的记忆此刻才生效（符合"不污染当前会话"铁律）
```

## 六、何时触发记忆存储（capture）

存储 = 把知识写入记忆仓库的 Markdown 文件。**触发主体是人或 Agent 主动调用 `capture`，
系统绝不自动写**（防止记忆变成幻觉垃圾堆）。

### 触发时机表

| 场景 | 触发 | 命令 |
|------|------|------|
| 做出重要技术决策（选型、架构） | ✓ 存储 | `capture -t decision "标题"` |
| 解决了一个非平凡错误（踩坑+解法） | ✓ 存储 | `capture -t error "标题"` |
| 验证了一个可复用流程 | ✓ 存储 | `capture -t skill "标题"` |
| 项目范围/架构发生明显变化 | ✓ 存储 | `capture -t project "标题"` |
| 发现重要的代码结构/坑点 | ✓ 存储 | `capture -t code "标题"` |
| 闲聊、临时调试、一次性脚本 | ✗ 不存储 | （留在工作记忆即可） |

### 存储动作的内部流程

```
team-memory capture "标题" -t decision --via codex --note "..."
   │
   ├─ 1. 定位记忆仓库（向上找 .team-memory.yml 标记）
   ├─ 2. 生成 id：mem-YYYYMMDD-NNN（扫描当天已有最大序号+1）
   ├─ 3. 套用类型模板，填入 title/source/created 等 frontmatter
   ├─ 4. 校验（必填字段、id 格式、status 一致性）
   ├─ 5. 写入 memory/{type}/mem-XXX.md
   └─ 6. 提示用户 git add/commit（不自动提交 —— 人工审核）
```

**关键设计**：
- `--via` 标注来源工具，让每条记忆可追溯"谁、用什么工具沉淀的"
- `--related` 关联其他记忆（如 Hermes 的选题关联 Claude 和 Codex 的项目）
- **不自动 commit**：写入文件后提示人工提交，保证记忆质量

## 七、何时触发记忆共享

共享 = 让记忆在**不同工具、不同人**之间可见。有三种共享动作，触发时机不同：

### 7.1 共享给本机的 AI 工具（`load --global`）

| 触发时机 | 动作 |
|----------|------|
| 初次接入 / 换了记忆仓库路径 | `load --global` 注册/更新 4 工具的全局 skill |
| 沉淀了新记忆，想让工具看到 | `load --global` 刷新 skill 里的记忆清单 |
| 新同事加入 | 他 clone 仓库后 `load --global` 一次 |

**效果**：在 `~/.claude/skills/`、`~/.hermes/skills/`、`~/.trae/skills/` 各装一个
`team-memory` skill；在 `~/.codex/AGENTS.md` 写入标记段。四个工具从此都能读到同一个仓库。
**未安装的工具自动跳过**。

### 7.2 共享给团队（`sync` → Gitea）

| 触发时机 | 动作 |
|----------|------|
| 本地沉淀了新记忆并 commit 后 | `sync` push 到 Gitea |
| 开始工作前 / 想拿队友最新记忆 | `sync` pull |
| 切换分支、合并冲突后 | `sync` 同步 |

**关键**：`sync` 只 pull/push **已 commit** 的内容，不自动 add/commit（人工审核）。
与现有的 `sync-remote-memory`（tencent-openclaw）**并存但不冲突**——本系统不重写它，
`sources.py` 把两者作为不同来源聚合读取。

### 7.3 查看/检索（`web` / `list` / `show`）

| 触发时机 | 动作 |
|----------|------|
| 想直观浏览所有记忆、看来源/版本 | `team-memory web`（浏览器可视化） |
| 终端快速扫一眼 | `team-memory list`（表格） |
| 看某条详情和版本演进 | `team-memory show mem-XXX` |

**Web UI 提供**：检索框（全字段）、来源彩色筛选、卡片视图、版本历史提交列表。

### 触发时机总览

```
沉淀新记忆 ──► load --global（让本机工具看到）
            │
            └──► git commit ──► sync（让团队看到）
                              │
                              └──► 队友 sync pull ──► load --global
                                                    （他们的工具也看到）

任何时候 ──► web / list / show（只读查看，不改记忆）
```

## 八、跨工具协作机制（核心价值）

以用户给的真实场景为例：

```
① Claude Code 沉淀：   "团队记忆系统采用纯Git" (source=claude, mem-001)
② Codex 沉淀：         "Think Tank Web 项目"   (source=codex,  mem-002)
③ Hermes 做公众号选题时：
   - 通过全局 skill 读到 mem-001 和 mem-002（跨工具共享生效）
   - 沉淀 "公众号选题可结合近期AI Native项目" (source=hermes, mem-003)
   - 用 --related 关联 mem-001, mem-002（建立跨工具引用）
```

**为什么能跨工具**：所有工具的 team-memory skill 都指向**同一个 Git 仓库**。
无论用哪个工具 `capture`，都写入同一处；无论用哪个工具，skill 都让它读到全部。
`source` 字段让你一眼看清每条记忆来自哪个工具，`list`/Web UI 用彩色 badge 可视化。

## 九、命令速查

| 命令 | 作用 | 触发什么 |
|------|------|----------|
| `team-memory init -p <路径>` | 建记忆仓库骨架 + git init | 一次性初始化 |
| `team-memory capture "标题" -t <类型> --via <工具>` | **存储**一条记忆 | 记忆存储触发 |
| `team-memory load --global` | 注册到 4 工具全局 skill | 记忆共享触发（本机） |
| `team-memory sync` | 与 Gitea pull/push | 记忆共享触发（团队） |
| `team-memory list` | 表格列出全部记忆 | 查看（来源/时间/版本） |
| `team-memory show <id>` | 单条详情 + 版本历史 | 查看 |
| `team-memory web` | 启动 Web 可视化 | 查看（检索/可视化） |

常用参数：`-p <仓库路径>`、`--via claude|codex|hermes|trae`、`--related <id,id>`、
`--note "简短正文"`、`--port <端口>`。

## 十、设计权衡与边界

### 选择"纯 Git"而非向量数据库（Phase 1）

- ✅ 人能读、AI 能读、可版本、可审核、可回滚、零外部依赖
- ⚠️ 检索靠关键字（substring + 多词 AND），记忆量大后检索弱
- 🔜 Phase 2 可叠加 mem0 向量层（MCP 只读），**Git 仍是唯一真相**

### 选择"人工审核"而非"自动写入"

- Agent 可以在对话里**建议**写记忆，但**不自动覆盖核心记忆**（project/decisions）
- 写入文件后提示 `git add/commit`，由人决定是否入库
- 这保证记忆是"团队共识的压缩包"，不是"AI 的临时理解"

### 安全边界

- `.env`（含 Gitea token）、`*.sqlite3`、向量库文件全部 `.gitignore`
- Codex 的 `AGENTS.md` 用 `<!-- team-memory:start/end -->` 标记段**幂等更新**，不破坏用户已有内容
- skill 写入是**新建 team-memory 子目录**，不动工具的其他 skill

## 十一、候选记忆 inbox + MCP 动态接入（v0.2 / v0.3 新增）

### 11.1 inbox 候选审核（Gateway 方案核心）
为防止 AI 错误经验污染正式记忆，新增**候选审核层**：
- AI / 未验证经验用 `propose` → 进 `inbox/`（status=pending_review），**不进正式 memory/**
- 人工 `review`（列候选）→ `approve`（转正式 active）/ `decline`（拒绝，留 inbox 供追溯）
- 候选带 `evidence`（PR / 会议 / bad case 来源）+ `confidence`（low/medium/high）便于审核

> 原则：**AI 可 propose，但不直接写正式记忆；approve 由人触发。**
> `capture`（人工确认）仍可直接进正式；`propose` 用于 AI 自动 / 存疑经验。

### 11.2 MCP Server（AI 工具动态读写）
让 AI 工具（Claude Code / Cursor / Codex / Hermes）通过 MCP 协议**动态调用**读写记忆，
不依赖 skill 静态快照或人跑 CLI：
- 启动：`pip install -e ".[mcp]"` → `python -m team_memory.mcp_server`
- 配 `TEAM_MEMORY_ROOT` 指向记忆仓库
- 6 个 tools：`search_memory` / `get_memory` / `list_memories` / `propose_memory`（候选）/ `list_inbox` / `approve_memory`
- 同样守"AI 可 propose，不直接写正式"原则

### 11.3 Gateway API（Web 升级为读写）
Web 不再只读，加写端点：`POST /api/propose`、`POST /api/approve/{id}`、`POST /api/decline/{id}`、`GET /api/inbox`。
浏览器 / curl / 其他服务都能读写。

### 11.4 状态机扩展（frontmatter `status`）

```
active           正式生效
pending_review   候选待审(inbox)
deprecated       已废弃
superseded       被新记忆取代(superseded_by 指向新)
declined         候选被拒绝(留 inbox 供追溯)
conflict         与其他记忆冲突
private          个人私有(不入团队共享)
```

## 附：快速开始

```bash
pip install -e ".[dev]"                              # 装工具
team-memory init -p ~/team-memory -n "你的团队"       # 建记忆仓库
team-memory capture "首条决策" -t decision --via claude  # 存第一条
team-memory load --global -p ~/team-memory           # 4 工具共享
team-memory web -p ~/team-memory                     # 浏览器看可视化
# 配好仓库 .env 的 GITEA_REMOTE_URL 后:
team-memory sync                                     # 推到 Gitea 让团队共享
```

测试：`pytest`（33 用例覆盖 models/store/init/capture/bridges/list）。
